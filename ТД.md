# Vervel — Техническое задание

> Фитнес-платформа для тренеров и атлетов с AI-функциями, аналитикой нагрузок и системой коммуникации.

---

## 0. Описание продукта

**Vervel** — связка инструментов для учёта тренировок, наглядной обратной связи по нагрузке (2D-аватар, зоны тела), планирования и общения тренер↔атлет, с опциональным AI и монетизацией AI через внутренний баланс.

### Роли

| Роль | Назначение |
|------|------------|
| `athlete` | Свои тренировки, аналитика, команда, чаты с тренером |
| `trainer` | Атлеты, группы, календарь, шаблоны, данные по атлетам |
| `both` | Обе роли; в UI переключается активный кабинет (тренер / атлет) |

### Атлет

- Календарь активности, детали тренировок, черновик тренировки.
- Создание тренировки вручную; **AI**: распознавание по фото, генерация по тексту, парсинг заметок и применение к тренировке.
- Экран **Avatar** — нагрузка и восстановление по зонам (модель с затуханием).
- **Аналитика** — период, зоны, графики.
- **Сила и прогресс** — единый экран силового журнала и дашборда упражнений (`/progression`); детали в **разделе 7**.
- **Периодизация** — ATL / CTL / TSB (PMC, Banister).
- **Серии и достижения** — streak, история, разблокировка и просмотр ачивок.
- **Команда** — группы и тренеры, инвайты (ссылка / QR), публичный профиль тренера.
- **Лидерборд** по группе.
- **Диалоги** — список чатов, сообщения, поток обновлений, отметка прочитанного.
- **Входящий видеозвонок** — опрос, приём/отклонение, LiveKit.
- **Профиль** — данные, фото, пароль, смена роли, замеры тела, тема, кошелёк AI и пополнение (ЮKassa), реферальная статистика.

### Тренер

- Обзор **«Сегодня»**, календарь и плановые тренировки (CRUD).
- **Команда** — атлеты и группы; карточка атлета (статы, аватар нагрузки, периодизация, тренировки и др.), **видеозвонок** 1:1 или группе.
- **Шаблоны** тренировок и **библиотека упражнений**.
- **Чаты** — личные и групповые.
- История/управление **видеозвонками** (API).

### Общее

- **Авторизация:** email/пароль (rate limit), OAuth VK и Яндекс, выбор роли для новых OAuth-пользователей; доступ по httpOnly cookie `auth_token` (на сервере подстановка в Bearer для guard).
- **AI:** баланс, транзакции, статус; тарификация чата по токенам.
- **Платежи:** пополнение кошелька, вебхук ЮKassa.
- **Web Push:** VAPID, подписка/отписка.
- **Обратная связь** с привязкой к пользователю.
- **PWA:** service worker (офлайн-страница, push).

### Публичная часть

- Лендинг, вход/регистрация, OAuth callback, приглашение по ссылке, документы (privacy / offer / seller).

## 6. Сервисы бизнес-логики

### WorkoutCalculator
Расчёт нагрузки по зонам тела при создании тренировки.

- **bodybuilding**: `weight × reps`, нормализуется к `[0..1]`
- **crossfit**: `rounds / MAX_ROUNDS_FOR_FULL_LOAD`
- **cardio**: `duration / MAX_TIME (3600с)`
- **calculateRecoveryState** — декаментное затухание нагрузки (λ=0.3) для модели восстановления
- **calculatePeriodStats** — агрегация за период: объём, интенсивность, breakdown по типам

### AiBalanceService
Кошелёк для AI-функций.

- `charge(userId, amount, description)` — списание с транзакцией БД
- `calculateChatCost(inputTokens, outputTokens)` — стоимость = `tokens/1000 × rate × markup`, мин. `CHAT_MIN_CHARGE`
- `InsufficientBalanceError` — бросается при нехватке средств

### YandexAiService
- `recognizeFromImage` — OCR (Yandex Vision) → LLM-парсинг → структурированная тренировка
- `generateFromText` — описание → тренировка через YandexGPT
- `chat` — история сообщений → ответ + `{ inputTokens, outputTokens }`

### StreakService
- Логика streak: тот же день → без изменений; +1 день → `currentStreak++`; пропуск → `currentStreak=1`
- Логирует каждое событие в `streak_history`
- После обновления вызывает `AchievementService.checkAndUnlockAchievements`

### AchievementService
Разблокировка достижений по триггерам:
- `streak_days` — серия N дней
- `total_workouts` — всего N тренировок
- `ai_chat_messages` — N сообщений AI-чата
- `groups_joined` — состоит в N группах
- `trainer_messages` — N сообщений тренерам

### ExerciseCatalog
Статический каталог. Fuzzy-матчинг (fuzzball `token_set_ratio ≥ 72`) для AI → каталог.

### ProgressionService
Силовой журнал, закрепления, **эталоны** (`user_exercise_standards` + алиасы), условный 1RM (формула Эпли), группировка по эталону или сырому `exerciseId`, обогащение названий (БД `exercises`, `ExerciseCatalog`, `capitalizeFirstForDisplay`). В payload каждой записи журнала — **`dashboardMetric`** (окна 30+30 дней, как раньше у отдельного дашборда). Детальное ТЗ — **раздел 7**.

## 7. Сила и прогресс (силовой журнал, эталоны, сводка по карточке)

Функциональность для атлета: наглядно видеть работу с весом по упражнениям, объединять разные id одного движения через **эталоны**, и видеть **краткую сводку 30+30 дней** прямо в карточке.

### 7.1. Продукт и навигация

- **Экран** `/progression` — **один** поток контента: встроенный **`StrengthLogScreen`** (`embedded`), без отдельной вкладки «Дашборд».
- Вход с **Аналитики**: блок быстрых ссылок «Сила и прогресс» → `/progression`.
- **Редиректы** (старые закладки): `/strength-log` и `/exercise-dashboard` → `/progression` (без `?tab=`).
- Вёрстка: `ProgressionHubScreen`, заголовок, **`SectionDivider`**, журнал.

### 7.2. Силовой журнал

**Источник данных**

- Тренировки пользователя за **около года** (`Workout`, не удалённые), упражнения с **хотя бы одним подходом с весом > 0** и непустыми сетами.
- Учитываются и **силовые**, и **WOD** (тип тренировки не отсекает запись).
- На упражнение показываются до **6 последних сессий** (по разным тренировкам).

**Закрепления (pins)**

- Список id хранится в таблице **`user_pinned_exercises`** (см. раздел 4).
- **Пустой список пинов** — в журнале отображаются **все** подходящие упражнения; API отдаёт **`suggestedPins`** — топ по числу сессий с весом (для кнопки «закрепить топ-N»).
- **Непустой список** — в журнале только закреплённые id; порядок карточек = порядок пинов.
- Добавить в пины можно из **истории за год** (все упражнения с весом), включая **`custom:…`** после ИИ — отдельный bottom sheet, не обзор каталога ~800 упражнений без истории.
- Важно: в режиме пинов карточка **не строится**, если по этому id **нет ни одной сессии с весом** в выборке (нет «пустых» карточек только из каталога).

**Карточка упражнения (UI)**

- Таблица: подходы × даты сессий, строка **1RM** (с бэкенда, Эпли по подходам сессии).
- Переключение **таблица / график / сводка 30 дней** (три режима); график — **Recharts** `LineChart`, ось Y — **best1RM** сессии или при отсутствии — **максимальный вес** в подходах.
- Режим **сводки** — поля из **`dashboardMetric`** в записи журнала (лучший 1RM за 30 дн., изменение % к предыдущим 30 дн., число сессий, дата последней тренировки).
- Кнопка **эталона** (sheet): создать эталон и привязать текущий сырой `exerciseId`, опционально **каталог** через `ExercisePicker`; для уже существующего эталона — переименование (PATCH) и добавление **синонимов** из списка упражнений с весом за год (ещё не занятых алиасом).
- Кастомный **тултип** точки графика (без переноса «кг»).
- Под названием — **прирост в %** от **самой ранней** к **самой поздней** сессии среди показанных (минимум две сессии с валидной метрикой); расчёт в `strengthLogChart.ts` (`strengthLogProgressPercent`), согласован с метрикой графика.

**Названия упражнений**

- Обогащение на бэкенде: `exercises.title` из БД, иначе заголовок из **`ExerciseCatalog`**, нормализация подчёркиваний, **`capitalizeFirstForDisplay`** для отображаемой первой буквы (в т.ч. если в JSON/тренировке строка со строчной).

### 7.3. Эталоны и метрика «как дашборд»

- Отдельной сущности «список упражнений дашборда» (**`user_dashboard_exercises`**) нет: сводка 30+30 считается **для каждой карточки журнала** (в т.ч. для группы эталона) и отдаётся в **`dashboardMetric`**.
- Объединение разных строк (`custom:…`, каталог, ИИ) — через **эталон** и алиасы `sourceExerciseId → standardId` (без цепочек).

### 7.4. Backend (реализация)

- **`ProgressionService`** (`apps/api/app/services/ProgressionService.ts`) — сбор журнала, пины, эталоны/алиасы, группировка, расчёт сессий, 1RM и **`dashboardMetric`** по записи.
- **`strength_log_support.ts`** — чистые функции для нормализации списка пинов и топа по частоте (юнит-тесты).
- **`progression_controller`** + роуты в `start/routes.ts` (см. раздел 5).
- Вспомогательные правила матчинга имён — **`exercise_match_helpers`**, каталог — **`ExerciseCatalog`**.

### 7.5. Frontend (реализация)

- `apps/web/src/screens/ProgressionHubScreen/ProgressionHubScreen.tsx`
- `StrengthLogScreen/`, `strengthLogChart.ts` (+ тесты Vitest для графика/процента).
- API-методы в `apps/web/src/api/athlete.ts` (`getStrengthLog`, `putStrengthLogPins`, CRUD эталонов и алиасов).

## 8. Модель периодизации (PMC — Banister)

```
TL  = (intensity×0.7 + min(volume/5000, 1)×0.3) × 100   — нагрузка дня
ATL = ATL_prev × (6/7) + TL × (1/7)    — 7-дн. EWMA  (усталость)
CTL = CTL_prev × (41/42) + TL × (1/42) — 42-дн. EWMA (форма)
TSB = CTL - ATL                          — свежесть
```

| TSB | Фаза |
|-----|------|
| > +15 | Deload / пик готовности |
| 0…+15 | Оптимальная форма |
| −20…0 | Накопление / рост формы |
| < −20 | Перетренированность |

## 10. Оплата (ЮКасса)

1. Пользователь выбирает сумму (100/250/500/1000₽)
2. `POST /payments/topup` → создаёт платёж ЮКасса → возвращает `confirmationUrl`
3. Редирект на страницу оплаты ЮКасса
4. После оплаты → `POST /payments/webhook` (IP-whitelist ЮКасса)
5. Backend: `balance += amount`, запись в `balance_transactions type='topup'`
6. Frontend: `/profile?topup=success` → toast + обновление баланса

## 11. AI-флоу создания тренировки

```
Фото/текст → YandexAiService → AiWorkoutResult {
  workoutType, exercises[{ name(EN), displayName(RU), sets, reps, weight }]
}
↓
ExerciseCatalog.fuzzyMatch(name) → exerciseId из каталога
↓
WorkoutInlineForm (редактирование перед сохранением)
↓
POST /workouts → WorkoutCalculator → сохранение + streak
```
