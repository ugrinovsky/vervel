# Vervel — повторный аудит прогресса (по спринтам 1–5)

Дата: 2026-05-06  
Основание: [VERVEL_AUDIT_CIS_2026.md](./VERVEL_AUDIT_CIS_2026.md), [VERVEL_SPRINT_PLAN_AUDIT_7_5.md](./VERVEL_SPRINT_PLAN_AUDIT_7_5.md)

Цель: кратко зафиксировать, что реально сделано по критическим red flags (1–3), какие новые операционные требования появились, и что остаётся до подтверждения “7.5/10”.

---

## 1) Red flags аудита — статус

### RF1 — Security debt (auth/oauth/payment)
**Статус:** существенно снижено (ключевые P0 закрыты).

**Закрыто по факту:**
- OAuth `state` (anti-CSRF) + корректные редиректы на `APP_URL`.
- `POST /oauth/set-role` теперь только для авторизованного пользователя (нельзя выставлять роль “чужому”).
- TTL access tokens (30 дней) синхронизирован с cookie TTL.
- Смена пароля отзывает все токены и выдаёт новый.
- Rate limits на OAuth endpoints.
- YooKassa webhook: идемпотентность с `FOR UPDATE`, сверки, верификация платежа через YooKassa API.

**Осталось / риск:**
- наблюдаемость security-событий (алерты/метрики), hardening секретов/ротация — частично вне текущего объёма.

### RF2 — Нет durable очередей
**Статус:** закрыто базовым устойчивым механизмом.

**Что сделано:**
- DB-backed очередь `jobs` + worker (poll) с `FOR UPDATE SKIP LOCKED` + retry/backoff.
- Мигрированы в очередь:
  - push-уведомления,
  - fan-out scheduled workouts (создание/синхронизация `Workout` для атлетов).

**Новый операционный риск (важно):**
- В production нужно включить воркер (`JOBS_WORKER_ENABLED=true`) и применять миграции (таблица `jobs`), иначе задачи будут копиться и не выполняться.

### RF3 — CSR-only + тяжёлый initial bundle
**Статус:** частично закрыто (быстрый выигрыш).

**Что сделано:**
- Route-level code splitting через `React.lazy` + `Suspense` на большинстве экранов и публичных роутов.
- Единый fallback (`RouteLoading`).
- Зафиксирован стартовый perf-бюджет: [VERVEL_WEB_PERF_BUDGET.md](./VERVEL_WEB_PERF_BUDGET.md).

**Осталось:**
- реальные замеры bundle/LCP и “budget enforcement” (CI или регламент релизов).
- SEO/SSR всё ещё остаётся как долгосрочный пункт.

### RF4 — Слишком широкий продукт до PMF
**Статус:** только зафиксирован документами (реальный продуктовый “режим PMF” пока не включали).

**Что сделано:**
- Зафиксированы PMF-scope и метрики: [VERVEL_PMF_SCOPE.md](./VERVEL_PMF_SCOPE.md), [VERVEL_METRICS_SPEC.md](./VERVEL_METRICS_SPEC.md).

**Осталось:**
- внедрение измерения метрик (события/панель).
- “PMF mode” по умолчанию в UX (это Спринт 6 — заморожен).

### RF5 — Нет fallback-плана по инфраструктуре
**Статус:** не закрыто (Спринт 6 заморожен).

---

## 2) Что поменялось в оценке “7.5/10”

- **Инженерные preconditions** для 7.5/10 стали ближе к факту: закрыты RF1–RF3 на базовом уровне и RF2 (очереди) не “на бумаге”.
- Но итоговая формулировка аудита (“7.5/10 при фокусе”) всё ещё упирается в:
  - **фокус GTM/PMF** (пока документально),
  - **измеримость воронки** (метрики/ивенты ещё не внедрены),
  - **fallback/runbook** (заморожено).

---

## 3) Новые требования к production (важно)

- Включить jobs worker в API:
  - `JOBS_WORKER_ENABLED=true`
  - `JOBS_POLL_INTERVAL_MS=2000` (или по нагрузке)
- Применить миграции (включая `jobs` таблицу).
- Убедиться, что API в production **не запускается в watch-режиме** и что `NODE_ENV=production`. В противном случае возможны перезапуски/нестабильная инициализация зависимостей (в т.ч. БД) под нагрузкой.

### Инцидент (май 2026): 502 Bad Gateway после деплоя

- Симптом: `502 Bad Gateway` на входе, в логах API — повторяющиеся ошибки воркера `Cannot connect to unregistered connection postgres`.
- Вывод: проблема была не в данных БД, а в **операционном запуске** (watch-режим/перезапуски + ранний доступ к БД из воркера), который сделал скрытую нестабильность очевидной.
- Митигирующие меры: зафиксировать production checklist и соблюдать его при деплоях (см. `ТД.md` §12).

---

## 4) Рекомендуемые следующие шаги (если “повторный аудит” приняли)

1) Завершить “доказательность” PMF: внедрить минимальный трекинг событий (из `VERVEL_METRICS_SPEC.md`) — без этого нельзя честно сказать “фокус сработал”.
2) Разморозить Спринт 6 хотя бы в части runbook/fallback (документ, не код) — это быстро и снижает RF5.
