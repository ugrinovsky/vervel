# Vervel Business Process Audit

Дата: 2026-04-26

## Цель

Собрать целостную карту:
1. Какие бизнес-процессы есть в системе.
2. Через какие интерфейсы и API они достигаются.
3. Где процессные разрывы и почему это бьет по activation/retention/монетизации.

---

## 1) Карта бизнес-процессов (Backend + UI)

| Процесс | Персона | Цепочка в UI | API-цепочка | Целевой результат | Основные риски |
|---|---|---|---|---|---|
| Auth (email) | Guest | Landing -> Login/Register -> Home | `POST /register`, `POST /login`, `POST /logout` | Пользователь вошел и попал в protected часть | Возможен drop на длинной регистрации |
| OAuth + выбор роли | Guest/OAuth user | OAuth callback -> SelectRole -> Home | `POST /oauth/*/sdk-login`, `GET /oauth/:provider/callback`, `POST /oauth/set-role` | Быстрый вход + корректная роль | Доп. шаг выбора роли повышает drop; `set-role` требует отдельного контроля |
| Привязка к тренеру через invite | Athlete | Invite link -> auth -> accept | `GET /invite/info/:token`, `POST /invite/accept` | Атлет связан с тренером/командой | Ошибка токена или auth-фрикция обрывает воронку |
| Профиль и настройки | User | Profile tab | `GET/PUT /profile`, `POST /profile/photo`, `PUT /profile/password` | Актуальные данные аккаунта | Низкий, mostly maintenance flow |
| Замеры тела (measurement loop) | User | Profile/Settings -> log measurement -> history | `POST/GET/DELETE /profile/measurements` | Отслеживание динамики тела и доп. retention-loop | Часто воспринимается как “второстепенно”, хотя это отдельный цикл возврата |
| Логирование тренировки | Athlete | Home/Calendar -> WorkoutForm -> Save | `POST/PUT/DELETE /workouts`, `GET /workouts/*`, draft endpoints | Регулярные записи тренировок | Сложность формы на первом входе |
| Черновики тренировок (draft recovery) | Athlete/Trainer | Начал заполнение -> вышел -> вернулся -> продолжил | Athlete: `GET/PUT/DELETE /workouts/draft`; Trainer: локальные draft в календаре | Доведение незавершенного ввода до сохранения | Недооценка влияет на activation и completion rate |
| Аналитика/аватар/прогресс | Athlete | Avatar/Analytics/Progression/StrengthLog | `GET /avatar/stats`, `GET /progression*`, `GET /athlete/periodization` | Понятный feedback loop прогресса | При неполных данных ценность падает |
| Streak/achievements | Athlete | Streak screen | `GET /streak*`, `PATCH /streak/mode`, achievements endpoints | Поведенческое удержание | Слабый эффект без регулярного триггера |
| Тренер: добавить атлета | Trainer | Team/Athletes -> Add drawer (email/invite/QR) | `POST /trainer/athletes/by-email`, `/invite`, `/by-qr` | Пополнение клиентской базы тренера | 3 метода в одном флоу = когнитивный шум |
| Тренер: группы и команда | Trainer | Groups/Team screens | `GET/POST/PUT/DELETE /trainer/groups*`, member endpoints | Управление группами клиентов | Переходы между экранами раздроблены |
| Тренер: планирование | Trainer | Calendar/Templates -> assign workouts | templates + scheduled endpoints | Регулярные назначения тренировок | Перегруженный календарный интерфейс |
| Тренер: intro-сессии (lead flow) | Trainer | Calendar -> intro slot (имя/телефон) | `POST /trainer/scheduled-workouts` (тип `intro`) | Доаккаунтное ведение потенциального клиента | Легко потерять из фокуса как отдельную воронку лидов |
| Тренер: автодоставка preview в чат | Trainer/Athlete | После назначения тренировки -> авто-сообщение в чат | `GET /trainer/chats/*`, `POST /trainer/chats/:chatId/messages` | Повышение adherence через коммуникацию | Без аналитики трудно связать эффект на retention |
| Чаты/диалоги | Athlete/Trainer | Dialogs/My Team/Detail screens | `/chats*`, `/trainer/chats*`, `/athlete/chats*`, SSE stream | Коммуникация и удержание | Дублированные точки входа, местами неочевидные переходы |
| Видеозвонки | Athlete/Trainer | Incoming/Call buttons | `/trainer/calls*`, `/calls/:roomName/join|decline`, active | Синхронная коммуникация | Зависимость от внешнего LiveKit |
| AI-функции + баланс | Athlete/Trainer | Workout AI widgets, AI chat, wallet | `/ai/*`, `/payments/topup`, webhook | Монетизация usage и ускорение ввода данных | Списание до результата в части AI сценариев |
| AI-анатомия кэш классификации | System (implicit) | Фоновый слой внутри AI/прогресс-фич | `ExerciseAnatomyService` + `exercise_anatomy_cache` | Стабильность зон, ниже latency/cost | Если игнорировать, растет стоимость и плавает качество |
| Донат тренеру (внешний) | Athlete/Trainer | TrainerPersonal: тренер задает реквизиты -> TrainerPublicProfile: атлет копирует СБП/карту или открывает ссылку ЮKassa | `GET/PUT /profile` (donate fields), `GET /athlete/trainers/:trainerId/profile` | Поддержка тренера прямым платежом вне внутреннего кошелька | Нет внутреннего трекинга факта доната/конверсии/суммы; часть флоу уходит вне системы |
| Push-уведомления | User | Browser permission + system notif | `/push/vapid-key`, `/push/subscribe`, `/push/unsubscribe` | Возврат пользователя в продукт | При неверном конфиге деградация “тихо” |
| Feedback intake | User | Profile/settings feedback form | `POST /feedback` | Канал обратной связи | Нет явного публичного lifecycle обработки |

## 2) Сквозные цепочки (как БП достигается)

## 2.1 Athlete activation chain (Day 1)
1. Регистрация/вход.
2. Выбор роли.
3. Первая тренировка.
4. Просмотр результата (календарь/аватар/прогресс).

**Где ломается:** обычно на шаге 3 (слишком сложный первый ввод) и между 3->4 (нет сильного “что дальше” триггера).

## 2.2 Trainer value chain (Day 1-7)
1. Вход как тренер.
2. Добавление атлетов.
3. Создание шаблона или тренировки.
4. Назначение на дату/группу.
5. Возврат по непрочитанным/отчетам.

**Где ломается:** длинная связка без wizard; много отдельных экранов.

## 2.3 Retention chain (Athlete)
1. Push/внутренний триггер.
2. Быстрый лог очередной тренировки.
3. Мгновенный feedback (серии, прогресс, аватар, рейтинг).
4. Следующая цель на неделю.

**Где ломается:** если нет четкого daily/weekly триггера и “следующего действия”.

## 2.4 Monetization chain (AI usage)
1. Пользователь открывает AI-фичу.
2. Проверяется баланс.
3. Списание.
4. Получение AI-результата.
5. Применение результата в тренировку.

**Где ломается:** если списание произошло, а ценность результата низкая/ошибка, trust падает.

## 2.5 Trainer donation chain (external payment)
1. Тренер в личном кабинете заполняет реквизиты поддержки (СБП/карта/ссылка ЮKassa).
2. Атлет открывает публичный профиль тренера.
3. Атлет выбирает сумму (опционально) и платит вне внутреннего кошелька.

**Где ломается:** система не видит факт оплаты end-to-end, поэтому нельзя полноценно измерять конверсию донатов и оптимизировать флоу по данным.

## 3) Проблемы по категориям

## A. Activation
- Первый сценарий перегружен количеством опций.
- Ролевой/инвайтный путь добавляет лишние точки выхода.
- Недостаточно “guided next step” после первой успешной тренировки.

## B. Retention
- Нет жесткого обязательного weekly loop (особенно без активного тренера).
- Соц-ценность сильная, но зависит от включенного trainer/group контура.
- Аналитика “не зажигает”, если данные вводятся неполно.

## C. Process integrity
- Есть цепочки, где backend может вернуть успех при частичной неудаче downstream-процесса.
- Некоторые процессы rely on best-effort async без гарантии завершения.

## D. Monetization
- AI-first модель реалистична, но чувствительна к:
  - качеству AI-результата,
  - частоте использования,
  - прозрачности списаний.
- Донаты тренеру уже есть, но это внешний процесс (реквизиты/ссылка), поэтому в текущем виде это скорее UX-фича поддержки, чем управляемый revenue pipeline с аналитикой.
- Без сильного retention AI-выручка может быть рваной.

## 4) Топ-10 разрывов “БП -> интерфейс”

1. **Длинный onboarding path** до “первая регулярная ценность”.
2. **Перегруженный WorkoutForm** на первом опыте.
3. **Отсутствие жесткого post-success сценария** после первой тренировки.
4. **Раздробленные точки входа в чат**, создают навигационный шум.
5. **Trainer planning flow** функционально мощный, но UI-тяжелый.
6. **Invite chain хрупкая к ошибкам токена/auth**.
7. **Непрозрачная деградация уведомлений** при плохой конфигурации.
8. **AI charge-first в части сценариев** ухудшает доверие при ошибке.
9. **Часть async-процессов best-effort**, без пользовательского feedback о реальном статусе.
10. **Сильная зависимость retention от поведения тренера**, если выбран social-driven путь.
11. **Донаты тренеру не трекаются внутри продукта** как завершенные транзакции, что мешает управлять этим каналом монетизации.

## 5) Что делать дальше (практика)

## P0 (1-2 недели)
- Сконцентрировать activation на одном сценарии: “быстро залогировал -> увидел прогресс”.
- Упростить first workout UX (минимум полей, step-by-step).
- Добавить обязательный “next action” после каждого сохранения.

## P1 (2-4 недели)
- Пересобрать trainer onboarding в единый guided flow.
- Унифицировать входы в чат и в командный контур.
- Ввести статусность для фоновых операций (не только “успех запроса”).

## P2 (4-6 недель)
- Усилить retention loop weekly summary + push cadence.
- Настроить честную AI-монетизацию: понятные списания, fallback-поведение на ошибках.
- Зафиксировать unit metrics по цепочке “действие -> возврат -> платное действие”.

---

## 6) KPI для проверки исправлений

- `Activation`: % пользователей с 1-й тренировкой в первые 24 часа.
- `Early retention`: % с 2+ тренировками за 7 дней.
- `Loop quality`: среднее время до следующего полезного действия после save.
- `Social retention`: % атлетов с активным чатом/командой в D7.
- `AI monetization`: доля пользователей с повторным AI-использованием в 7 дней после первой оплаты.

---

## Итог

Да, система уже содержит все ключевые бизнес-процессы (не “MVP-пустышка”), но слабое место сейчас не в количестве фич, а в **склейке процессов в короткие повторяемые цепочки**.

Если закрыть разрывы в activation/retention и процессной целостности, текущий функционал начнет работать как система, а не как набор возможностей.
