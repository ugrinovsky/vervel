# План спринтов: условия оценки 7,5/10 (VERVEL_AUDIT_CIS_2026)

Дата фиксации: 2026-05-06  
Основание: [VERVEL_AUDIT_CIS_2026.md](./VERVEL_AUDIT_CIS_2026.md) — **7,5/10** при **фокусе GTM/продукта** и **закрытии критических рисков**; иначе в аудите **4/10**.

Длина спринта: **2 недели** (при необходимости скорректировать под команду).

Связанные документы: [VERVEL_PMF_FOCUS_PLAN.md](./VERVEL_PMF_FOCUS_PLAN.md), [ONBOARDING_ADAPTIVE_UX.md](../ONBOARDING_ADAPTIVE_UX.md) (онбординг уже двигает активацию, но **не снимает** red flags из аудита).

## Цель плана

Довести продукт и инженерию до состояния, при котором формулировка аудита «фокус + закрытые критические риски» становится **правдой по фактам**, а не намерению.

---

## Спринт 1 — Фокус: ядро PMF и границы

**Цель:** зафиксировать «операционка тренера + команда» как **главный** нарратив и **границы ядра** для ближайших кварталов.

**Артефакты спринта (зафиксировано в докс):**
- [VERVEL_PMF_SCOPE.md](./VERVEL_PMF_SCOPE.md)
- [VERVEL_METRICS_SPEC.md](./VERVEL_METRICS_SPEC.md)

| Результат | Критерий готовности (Definition of Done) |
|-----------|------------------------------------------|
| Одностраничный **PMF scope** (что продаём тренеру первым, что — вторым) | Согласовано с владельцем продукта; есть список **не делаем сейчас** |
| **Метрики владельцы** | Назначены ответственные за: TTFA (time-to-first-action), инвайт → принятие, активный тренер (≥1 атлет), D7 тренера |
| Маркетинг/лендинг | Черновик сообщений под B2B2C из PMF Focus Plan; нет противоречия «приложение для всех сразу» |

**Не обязательно:** тяжёлый рефактор кода — достаточно решений и бэклога.

## Спринт 2 — Безопасность: периметр auth и OAuth

**Цель:** снять **red flag #1** (долг вокруг auth/oauth) до измеримого минимума.

| Результат | Критерий готовности |
|-----------|---------------------|
| Инвентаризация поверхности | Список эндпоинтов и потоков: сессия, refresh, OAuth callback, выход, сброс пароля |
| Устранение **P0/P1** | Закрыты найденные критические зазоры (утечки токенов, небезопасные редиректы, отсутствие state/nonce где нужно — по факту аудита кода) |
| Документация | Короткий **security notes** для команды: что нельзя ломать при фичах |

**Чеклист прод-конфига (auth/OAuth/cookies/CORS):**
- [VERVEL_AUTH_OAUTH_PROD_CHECKLIST.md](./VERVEL_AUTH_OAUTH_PROD_CHECKLIST.md)

**Сделано (май 2026, зафиксировано в коде):**
- OAuth `state` (anti-CSRF): генерация, cookie, проверка в callback.
- Редиректы после OAuth callback возвращают на **web (`APP_URL`)**, а не на API-домен.
- Раздельный парсинг token responses: VK vs Yandex (убран VK-парсер для Яндекса).
- `POST /oauth/set-role` защищён: требует auth и запрещает ставить роль за другого пользователя.
- Access tokens имеют явный TTL: `expiresIn: '30 days'` (синхронизировано с cookie TTL).
- При смене пароля: ревок всех токенов пользователя + выдача нового токена (не оставляем “утёкшие” токены живыми).
- Rate limiting на OAuth endpoints (redirect/callback + sdk/miniapplogin) для защиты от брутфорса/нагрузки.

## Спринт 3 — Безопасность: платежи и чувствительные данные

**Цель:** добить периметр **оплаты** и обработки платёжных/персональных данных в духе red flag #1.

| Результат | Критерий готовности |
|-----------|---------------------|
| Обзор ЮKassa и webhook | Идемпотентность, проверка подписи/секретов, обработка ошибок |
| Секреты и конфиг | Нет секретов в клиенте; ротация и доступы описаны |
| Логирование | Нет утечки PAN/токенов в логи |

**Сделано (май 2026, зафиксировано в коде):**
- Webhook `payment.succeeded`: защита от гонок (строка `payments` блокируется `FOR UPDATE`), начисление баланса строго один раз.
- Webhook: перед начислением выполняется **верификация платежа через YooKassa API** (`GET /v3/payments/{id}`), как рекомендует документация (проверка актуального статуса/объекта).
- Webhook: сверка суммы и `metadata.user_id` с записью в `payments` перед начислением (anti-tamper).
- Добавлен load-protection rate limit на `/payments/webhook`.

## Спринт 4 — Надёжность: durable-очереди и фоновые задачи

**Цель:** снять **red flag #2** — критичная фоновая работа не «best effort».

| Результат | Критерий готовности |
|-----------|---------------------|
| Очередь с персистенцией | Выбран и внедрён механизм (например Redis + очередь), **retry** и **dead-letter** для критичных job |
| Миграция | Минимум один класс критичных операций переведён с «fire-and-forget» на очередь |
| Наблюдаемость | Метрики/алерты на рост failed jobs |

---

## Спринт 5 — Производительность фронта и SEO-риск

**Цель:** ослабить **red flag #3** (тяжёлый initial bundle, CSR).

| Результат | Критерий готовности |
|-----------|---------------------|
| **Route-level code splitting** | Ленивая загрузка для крупных зон (тренерский кабинет, тяжёлые экраны) — измеримо снижен initial JS |
| Бюджет | Зафиксирован лимит на главный чанк / LCP-цель; регрессия ловится в CI или ручным чеклистом |
| Документ | Заметка «как добавлять экран без раздува бандла» |

*SSR/SSG в аудите как долгосрочный вектор; в этом спринте — реалистичный минимум для мобильного первого входа.*

---

## Спринт 6 — «Ширина продукта» и инфраструктурный fallback

**Цель:** red flags **#4** и **#5** — осознанное сужение до PMF и план устойчивости.

| Результат | Критерий готовности |
|-----------|---------------------|
| Сужение для новых пользователей | По PMF scope: умолчания, навигация и **feat-флаги** ведут тренера в «ядро»; лишнее не в лицо на день 1 |
| **Runbook** инфраструктуры | Кто звонит кому при падении провайдера; что отключаем; RPO/RTO на уровне «честно описано» |
| Fallback | Для 1–2 критичных внешних зависимостей — задокументирован запасной сценарий или деградация |
